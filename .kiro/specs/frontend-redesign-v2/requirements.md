# Requirements Document: Frontend Redesign V2

## Introduction

This document specifies the requirements for a major frontend redesign of the AI Recipe Generator application. The redesign introduces animated visual elements, custom cursor interactions, comprehensive user profile management, first-time onboarding, expanded recipe input controls, and enhanced Bedrock prompt engineering. The goal is to create an engaging, personalized, and feature-rich user experience while maintaining the existing backend infrastructure.

## Glossary

- **Frontend_Application**: The React/Next.js web application hosted on AWS Amplify
- **User_Profile**: A DynamoDB record containing user preferences, settings, and metadata
- **Recipe_History**: A collection of all recipes generated by a specific user stored in DynamoDB
- **Onboarding_Quiz**: A 7-step modal workflow presented to first-time users after sign-up
- **Floating_Elements**: SVG icons that animate continuously across the background using CSS keyframes
- **Custom_Cursor**: A frying pan SVG icon that replaces the default browser cursor
- **Recipe_Form**: The comprehensive input interface for recipe generation parameters
- **Bedrock_Prompt**: The dynamically constructed text sent to Amazon Bedrock for recipe generation
- **DynamoDB_Users_Table**: Database table storing user profile information
- **DynamoDB_RecipeHistory_Table**: Database table storing generated recipes with user associations
- **Cognito_First_Login_Flag**: A user attribute indicating whether the user has completed onboarding

## Requirements

### Requirement 1: Animated Floating Food Elements

**User Story:** As a user, I want to see animated food-related icons floating across the background, so that the interface feels more engaging and thematic.

#### Acceptance Criteria

1. THE Frontend_Application SHALL display 10-15 floating SVG food icons across the background
2. THE Frontend_Application SHALL animate floating elements using CSS keyframes with continuous movement
3. WHEN rendering floating elements, THE Frontend_Application SHALL vary animation speed between 8 and 20 seconds
4. WHEN rendering floating elements, THE Frontend_Application SHALL apply random delays to create staggered motion
5. WHEN rendering floating elements, THE Frontend_Application SHALL apply rotation between -15 degrees and +15 degrees
6. THE Frontend_Application SHALL set floating element opacity between 0.06 and 0.12 for subtle visual effect
7. THE Frontend_Application SHALL include food-related SVG icons such as vegetables, cookware, and utensils
8. WHEN a user hovers over cards, buttons, or panels, THE Frontend_Application SHALL apply translateY, scale, and border transitions

### Requirement 2: Custom Cursor Implementation

**User Story:** As a user, I want to see a custom frying pan cursor instead of the default pointer, so that the interface feels more immersive and thematic.

#### Acceptance Criteria

1. THE Frontend_Application SHALL replace the default cursor with a frying pan SVG icon sized at 28x28 pixels
2. WHEN a user hovers over interactive elements, THE Frontend_Application SHALL display a frying pan with sizzle or steam animation
3. THE Frontend_Application SHALL implement the custom cursor using CSS cursor: url() with base64-encoded SVG
4. IF the custom cursor fails to load, THEN THE Frontend_Application SHALL fallback to cursor: pointer
5. THE Frontend_Application SHALL ensure the custom cursor is visible across all pages and components

### Requirement 3: User Profile Management

**User Story:** As a user, I want to view and edit my profile with preferences and dietary information, so that I can personalize my recipe generation experience.

#### Acceptance Criteria

1. THE Frontend_Application SHALL display a profile page with avatar, display name, email, and preferences
2. THE Frontend_Application SHALL display user preferences including cuisine preference, dietary type, spice level, and health goal
3. WHEN a user clicks edit profile, THE Frontend_Application SHALL allow modification of display name and all preference fields
4. WHEN a user saves profile changes, THE Frontend_Application SHALL update the DynamoDB_Users_Table with new values
5. THE Frontend_Application SHALL retrieve user profile data from DynamoDB_Users_Table on profile page load
6. THE Frontend_Application SHALL validate profile data before saving to ensure required fields are populated

### Requirement 4: Recipe History Panel

**User Story:** As a user, I want to view all my previously generated recipes in one place, so that I can revisit and manage my recipe collection.

#### Acceptance Criteria

1. THE Frontend_Application SHALL display a recipe history panel listing all user recipes from DynamoDB_RecipeHistory_Table
2. WHEN rendering recipe history, THE Frontend_Application SHALL display each recipe as a card with title, date, and thumbnail
3. WHEN a user clicks a recipe card, THE Frontend_Application SHALL display the full recipe details
4. THE Frontend_Application SHALL provide a "View recipe" button on each recipe card
5. WHEN a user clicks the favorite toggle, THE Frontend_Application SHALL update the recipe favorite status in DynamoDB_RecipeHistory_Table
6. THE Frontend_Application SHALL display a star icon to indicate favorite status on recipe cards
7. THE Frontend_Application SHALL provide filter options for cuisine type and meal type
8. WHEN a user applies filters, THE Frontend_Application SHALL display only recipes matching the selected criteria
9. WHEN a user clicks delete on a recipe, THE Frontend_Application SHALL remove the recipe from DynamoDB_RecipeHistory_Table
10. WHEN a user deletes a recipe, THE Frontend_Application SHALL prompt for confirmation before deletion

### Requirement 5: First-Time Onboarding Quiz

**User Story:** As a new user, I want to complete an onboarding quiz after sign-up, so that the system can learn my preferences and provide personalized recipe suggestions.

#### Acceptance Criteria

1. WHEN a user signs up for the first time, THE Frontend_Application SHALL display a full-screen onboarding modal
2. THE Frontend_Application SHALL prevent skipping the onboarding quiz on first use
3. THE Frontend_Application SHALL present Step 1 asking for diet preference with options: Vegetarian, Non-Vegetarian, Vegan, Eggetarian
4. THE Frontend_Application SHALL present Step 2 asking for spice level with options: Mild, Medium, Spicy, Very Spicy
5. THE Frontend_Application SHALL present Step 3 asking for cooking goal with options: Taste, Fitness, Quick, Balanced
6. THE Frontend_Application SHALL present Step 4 asking for favorite cuisines with multi-select capability
7. THE Frontend_Application SHALL present Step 5 asking for available appliances with multi-select icon interface
8. THE Frontend_Application SHALL present Step 6 asking for dietary restrictions and allergies with multi-select capability
9. THE Frontend_Application SHALL present Step 7 asking for usual cooking time with options: Under 15min, 15-30min, 30-60min, Over 1hr
10. WHEN a user completes the onboarding quiz, THE Frontend_Application SHALL store all responses in DynamoDB_Users_Table
11. WHEN a user completes the onboarding quiz, THE Frontend_Application SHALL update the Cognito_First_Login_Flag to indicate completion
12. WHEN a user opens the recipe form, THE Frontend_Application SHALL pre-populate fields with onboarding preferences as defaults

### Requirement 6: Comprehensive Recipe Input Form

**User Story:** As a user, I want to specify detailed recipe parameters including diet type, meal type, cuisine, cooking style, and more, so that I can generate recipes that match my exact needs.

#### Acceptance Criteria

1. THE Recipe_Form SHALL provide a diet type toggle with options: Vegetarian, Non-Vegetarian, Vegan, Eggetarian
2. THE Recipe_Form SHALL provide a meal type selector with options: Breakfast, Lunch, Snack, Dinner, Dessert
3. THE Recipe_Form SHALL provide a cuisine selector with scrollable chips displaying 15 or more cuisine options
4. THE Recipe_Form SHALL provide a cooking style selector with options: Curry, Dry, Deep fry, Semi-curry, Steamed, Baked, Grilled, Salad
5. THE Recipe_Form SHALL provide an available appliances multi-select interface with icon representations
6. THE Recipe_Form SHALL provide pantry staples toggles with common items defaulted to ON
7. THE Recipe_Form SHALL provide a spice level slider with options: Mild, Medium, Spicy, Extra spicy
8. THE Recipe_Form SHALL provide a health goal selector with options: Fitness, Taste, Balanced
9. THE Recipe_Form SHALL provide a cook time slider ranging from 10 to 120 minutes with 5-minute increments
10. THE Recipe_Form SHALL provide an ingredient input field accepting free text with comma-separated values
11. THE Recipe_Form SHALL provide an optional voice input button using Web Speech API
12. WHEN a user clicks the voice input button, THE Recipe_Form SHALL capture spoken ingredients and populate the ingredient field
13. THE Recipe_Form SHALL display a large "Generate my recipe" button at the bottom of the form
14. WHEN a user has completed onboarding, THE Recipe_Form SHALL pre-populate fields with user preferences from DynamoDB_Users_Table

### Requirement 7: Enhanced Bedrock Prompt Engineering

**User Story:** As a user, I want the AI to consider all my specified parameters when generating recipes, so that the results are highly personalized and relevant.

#### Acceptance Criteria

1. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including diet type
2. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including meal type
3. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including cuisine preference
4. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including cooking style
5. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including available appliances
6. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including pantry staples
7. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including spice level
8. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including health goal
9. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including cook time constraint
10. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including user-provided ingredients
11. WHEN generating a recipe, THE Frontend_Application SHALL construct a Bedrock_Prompt including dietary restrictions from user profile
12. WHEN sending the prompt to Bedrock, THE Frontend_Application SHALL request recipe name, description, ingredients with quantities, step-by-step instructions, nutrition information, difficulty level, and 2 variation tips

### Requirement 8: DynamoDB Schema for User Profiles

**User Story:** As a developer, I want a well-defined database schema for user profiles, so that user preferences and settings are stored consistently.

#### Acceptance Criteria

1. THE DynamoDB_Users_Table SHALL have a primary key of userId (string)
2. THE DynamoDB_Users_Table SHALL store displayName (string)
3. THE DynamoDB_Users_Table SHALL store email (string)
4. THE DynamoDB_Users_Table SHALL store dietPreference (string)
5. THE DynamoDB_Users_Table SHALL store spiceLevel (string)
6. THE DynamoDB_Users_Table SHALL store cookingGoal (string)
7. THE DynamoDB_Users_Table SHALL store favoriteCuisines (string array)
8. THE DynamoDB_Users_Table SHALL store availableAppliances (string array)
9. THE DynamoDB_Users_Table SHALL store dietaryRestrictions (string array)
10. THE DynamoDB_Users_Table SHALL store usualCookingTime (string)
11. THE DynamoDB_Users_Table SHALL store hasCompletedOnboarding (boolean)
12. THE DynamoDB_Users_Table SHALL store createdAt (timestamp)
13. THE DynamoDB_Users_Table SHALL store updatedAt (timestamp)

### Requirement 9: DynamoDB Schema for Recipe History

**User Story:** As a developer, I want a well-defined database schema for recipe history, so that user recipes are stored with proper associations and metadata.

#### Acceptance Criteria

1. THE DynamoDB_RecipeHistory_Table SHALL have a composite primary key with userId (partition key) and recipeId (sort key)
2. THE DynamoDB_RecipeHistory_Table SHALL store recipeName (string)
3. THE DynamoDB_RecipeHistory_Table SHALL store recipeDescription (string)
4. THE DynamoDB_RecipeHistory_Table SHALL store ingredients (list of objects with name, amount, unit)
5. THE DynamoDB_RecipeHistory_Table SHALL store instructions (string array)
6. THE DynamoDB_RecipeHistory_Table SHALL store nutritionInfo (object with calories, protein, carbs, fat)
7. THE DynamoDB_RecipeHistory_Table SHALL store difficulty (string)
8. THE DynamoDB_RecipeHistory_Table SHALL store variationTips (string array)
9. THE DynamoDB_RecipeHistory_Table SHALL store cuisine (string)
10. THE DynamoDB_RecipeHistory_Table SHALL store mealType (string)
11. THE DynamoDB_RecipeHistory_Table SHALL store cookTime (number)
12. THE DynamoDB_RecipeHistory_Table SHALL store isFavorite (boolean)
13. THE DynamoDB_RecipeHistory_Table SHALL store createdAt (timestamp)

### Requirement 10: Lambda Functions for Profile Management

**User Story:** As a developer, I want Lambda functions to handle profile CRUD operations, so that the frontend can interact with user profile data securely.

#### Acceptance Criteria

1. THE Backend SHALL provide a Lambda function to retrieve user profile from DynamoDB_Users_Table
2. THE Backend SHALL provide a Lambda function to update user profile in DynamoDB_Users_Table
3. THE Backend SHALL provide a Lambda function to create initial user profile after onboarding
4. WHEN a Lambda function receives a profile request, THE Backend SHALL validate the JWT token from Cognito
5. WHEN a Lambda function updates a profile, THE Backend SHALL update the updatedAt timestamp
6. IF a profile update fails, THEN THE Backend SHALL return a descriptive error message
7. THE Backend SHALL ensure users can only access their own profile data

### Requirement 11: Lambda Functions for Recipe History Management

**User Story:** As a developer, I want Lambda functions to handle recipe history operations, so that users can retrieve, favorite, and delete their recipes.

#### Acceptance Criteria

1. THE Backend SHALL provide a Lambda function to retrieve all recipes for a user from DynamoDB_RecipeHistory_Table
2. THE Backend SHALL provide a Lambda function to retrieve a single recipe by recipeId
3. THE Backend SHALL provide a Lambda function to update recipe favorite status
4. THE Backend SHALL provide a Lambda function to delete a recipe from DynamoDB_RecipeHistory_Table
5. WHEN a Lambda function receives a recipe history request, THE Backend SHALL validate the JWT token from Cognito
6. WHEN retrieving recipes, THE Backend SHALL support filtering by cuisine and mealType
7. WHEN retrieving recipes, THE Backend SHALL sort results by createdAt in descending order
8. THE Backend SHALL ensure users can only access their own recipe history

### Requirement 12: Responsive Design for All New Features

**User Story:** As a user, I want all new features to work seamlessly on both mobile and desktop devices, so that I can use the application on any device.

#### Acceptance Criteria

1. THE Frontend_Application SHALL render the onboarding quiz responsively on screen sizes from 320px to 1920px width
2. THE Frontend_Application SHALL render the recipe form responsively on screen sizes from 320px to 1920px width
3. THE Frontend_Application SHALL render the profile page responsively on screen sizes from 320px to 1920px width
4. THE Frontend_Application SHALL render the recipe history panel responsively on screen sizes from 320px to 1920px width
5. WHEN viewed on mobile devices, THE Frontend_Application SHALL stack form controls vertically
6. WHEN viewed on desktop devices, THE Frontend_Application SHALL arrange form controls in a multi-column layout
7. THE Frontend_Application SHALL ensure floating elements do not interfere with touch interactions on mobile devices
8. THE Frontend_Application SHALL ensure the custom cursor is disabled on touch devices

### Requirement 13: Voice Input for Ingredients

**User Story:** As a user, I want to speak my ingredients instead of typing them, so that I can input ingredients more quickly and conveniently.

#### Acceptance Criteria

1. THE Recipe_Form SHALL provide a microphone button next to the ingredient input field
2. WHEN a user clicks the microphone button, THE Recipe_Form SHALL request microphone permission
3. IF microphone permission is granted, THEN THE Recipe_Form SHALL activate the Web Speech API
4. WHEN the Web Speech API is active, THE Recipe_Form SHALL capture spoken words and convert them to text
5. WHEN speech recognition completes, THE Recipe_Form SHALL populate the ingredient field with recognized text
6. IF the Web Speech API is not supported, THEN THE Recipe_Form SHALL hide the microphone button
7. WHEN speech recognition fails, THE Recipe_Form SHALL display an error message to the user

### Requirement 14: Local Development Environment

**User Story:** As a developer, I want to run the redesigned frontend locally before deployment, so that I can test and debug features in a controlled environment.

#### Acceptance Criteria

1. THE Frontend_Application SHALL run locally using npm run dev or equivalent command
2. WHEN running locally, THE Frontend_Application SHALL connect to existing AWS backend services
3. WHEN running locally, THE Frontend_Application SHALL use environment variables for API endpoints
4. THE Frontend_Application SHALL provide clear documentation for local setup and configuration
5. THE Frontend_Application SHALL not require AWS deployment for initial testing
6. THE Frontend_Application SHALL not push code to GitHub until explicitly requested by the user

### Requirement 15: CSS-Only Animations

**User Story:** As a developer, I want all animations to use pure CSS without JavaScript libraries, so that performance is optimized and dependencies are minimized.

#### Acceptance Criteria

1. THE Frontend_Application SHALL implement floating element animations using CSS @keyframes
2. THE Frontend_Application SHALL implement hover effects using CSS transitions
3. THE Frontend_Application SHALL implement custom cursor using CSS cursor property with base64 SVG
4. THE Frontend_Application SHALL not use JavaScript animation libraries for visual effects
5. THE Frontend_Application SHALL ensure animations are hardware-accelerated using transform and opacity properties
6. THE Frontend_Application SHALL provide fallback styles for browsers that do not support CSS animations

### Requirement 16: Parser and Serializer for Recipe Data

**User Story:** As a developer, I want reliable parsing and serialization of recipe data between frontend and backend, so that data integrity is maintained throughout the system.

#### Acceptance Criteria

1. WHEN receiving recipe data from Bedrock, THE Backend SHALL parse the JSON response into a Recipe object
2. WHEN parsing fails, THE Backend SHALL return a descriptive error message
3. THE Backend SHALL provide a pretty printer to format Recipe objects into JSON
4. FOR ALL valid Recipe objects, parsing then printing then parsing SHALL produce an equivalent object (round-trip property)
5. THE Frontend_Application SHALL parse recipe JSON from API responses into typed Recipe objects
6. THE Frontend_Application SHALL serialize recipe filter parameters into query strings for API requests
7. IF recipe data is malformed, THEN THE Frontend_Application SHALL display a user-friendly error message
